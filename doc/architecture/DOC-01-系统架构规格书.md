## 1. 架构总览

### 1.1 系统定位

IntelliServe IT Suite 是一个**面向企业内网落地的 AI 驱动 IT 运维平台**，面向大型园区（2000-5000 台终端规模），融合大语言模型（LLM）、知识检索增强生成（RAG）、自动化脚本执行、资产全生命周期管理，实现从"被动救火"到"主动预防"的运维模式转型。系统支持开发期通过阿里云百炼调用 DeepSeek-V4 API 快速验证模型效果，也支持正式生产阶段基于 DeepSeek-V4 开源权重切换到企业私有化 GPU 推理集群或专属模型网关。

### 1.2 核心设计原则

| 原则 | 说明 |
|------|------|
| **企业可控优先** | 数据存储和业务逻辑默认内网部署；开发期优先使用百炼 DeepSeek-V4 API 加速验证，生产期按合规要求切换专线 API、专属实例或 DeepSeek-V4 私有化 GPU 集群 |
| **模块化单体 + Proxy 扩展** | 核心服务模块化单体部署，通过 Proxy 层水平扩展至多 VLAN 分区 |
| **异步优先** | 长耗时操作（脚本执行、AI 诊断、批量 OCR）一律通过任务队列异步处理 |
| **API 优先** | 所有功能通过 REST API 暴露，前端和 IM 机器人为 API 消费者 |
| **渐进式智能** | AI 能力分阶段上线：规则 → RAG → ML 预测，每阶段可控可回退 |
| **容器化交付** | 全部服务 Docker 化，`docker compose up -d` 一键部署 |

## 2. 四层架构详解

### 2.1 感知采集层（Perception）

**职责**：全量采集 IT 环境中的所有运维数据。

| 数据源 | 采集方式 | 存储目标 | 采集频率 |
|--------|---------|---------|---------|
| CPU/内存/磁盘/温度 | Zabbix Agent 2 (主动模式) | TimescaleDB (via PostgreSQL) | 60s |
| Windows Event Log | Zabbix Agent 2 `eventlog` 键 + Filebeat | TimescaleDB + MinIO (归档) | 实时 |
| 资产台账 | Web 控制台手动录入 + OCR 自动录入 | PostgreSQL | 按需 |
| 网络设备 SNMP | Zabbix SNMP 模板 | PostgreSQL/TimescaleDB | 300s |
| 用户工单/聊天 | 钉钉 Stream Mode / 企业微信 Webhook | PostgreSQL | 实时 |
| 软件安装清单 | Zabbix Agent 2 `software` 键 | PostgreSQL | 3600s |

**关键设计决策**：
- Zabbix Agent 2 作为主采集代理（Go 重写版，15-30MB 内存），覆盖 Windows/macOS 终端基础采集需求
- Phase 3 引入自研轻量 Agent（Go），融合本地 LLM 诊断能力与 Windows/macOS 平台适配层
- 多 VLAN 分区通过 Zabbix Proxy / 自研 Proxy 中转上报
- 所有时序数据统一走 TimescaleDB 超表，避免 InfluxDB 额外运维

### 2.2 智能决策分析层（Decision）

**职责**：对采集数据进行智能分析，产出诊断结论与行动建议。

核心组件：

```
                   ┌─────────────┐
                   │   API 请求   │
                   └──────┬──────┘
                          │
              ┌───────────▼───────────┐
              │     意图路由器         │
              │  (Intent Router)      │
              └───┬───────┬───────┬──┘
                  │       │       │
        ┌─────────▼──┐ ┌──▼────┐ ┌▼─────────┐
        │  L1: RAG   │ │ L2:   │ │ L3:      │
        │  知识库问答 │ │ 自动化 │ │ 工单升级 │
        │            │ │ 诊断   │ │          │
        └──────┬─────┘ └──┬────┘ └────┬─────┘
               │          │           │
        ┌──────▼──────┐   │    ┌──────▼──────┐
        │ Qdrant 向量 │   │    │ 工单系统     │
        │ 语义检索     │   │    │ + 工程师指派 │
        └──────┬──────┘   │    └─────────────┘
               │          │
        ┌──────▼──────────▼──────┐
        │    LLM Provider 网关    │
        │ DashScope / 私有化模型  │
        └────────────────────────┘
```

**L1 — 知识库 RAG 问答**：
1. 用户问题 → LLMClient 嵌入（开发期 `text-embedding-v4` 默认 1024 维，生产私有化可切换 `BAAI/bge-m3` 1024 维；若任一模型降维需另建集合）→ Qdrant 语义检索 Top-5 知识库片段
2. 检索结果 + 用户问题 → 拼接 Prompt → LLM Provider 生成回答
3. 置信度评估：若 LLM 返回低置信度标记 → 升级至 L2

**L2 — 自动化诊断与修复**：
1. 采集终端系统日志、错误码、硬件状态
2. 拼接诊断 Prompt → LLM 分析 → 输出 JSON：`{root_cause, confidence, suggested_actions, script_id}`
3. 匹配合适的自动化脚本 → 触发 Celery 任务执行
4. 置信度不足 → 升级至 L3

**L3 — 工单升级与工程师指派**：
1. 汇总诊断上下文 → 生成结构化诊断报告
2. 创建工单 → AI 推荐工程师（基于技能标签匹配 + 当前负载）
3. 推送企业微信/钉钉通知 → 工程师接单

### 2.3 自动化执行层（Execution）

**职责**：将决策层的指令转化为实际操作。

| 执行能力 | 技术实现 | 触发方式 |
|---------|---------|---------|
| PowerShell 脚本执行 | Celery Worker → WinRM/PowerShell Remoting → 目标终端 | 用户自助 / AI 自动 / 工程师手动 |
| 网络配置重置 | `network-reset.ps1`：ipconfig /release /renew, flush DNS, Winsock reset | L1/L2 自动 |
| Office 组件修复 | `office-repair.ps1`：调用 Office 修复工具 | L1 自动 |
| 系统缓存清理 | `cache-cleanup.ps1`：清理 Temp、%TEMP%、浏览器缓存、Windows Update 缓存 | L1 自动 |
| 打印机故障修复 | `printer-reset.ps1`：重启 Spooler 服务，清除打印队列 | L1 自动 |
| 磁盘映射修复 | `drive-remap.ps1`：重新连接网络驱动器 | L1 自动 |
| 批量软件部署 | Ansible (Phase 2+) / PowerShell 脚本 | 工程师触发 |
| 合规自动审计 | Celery Beat 定时任务：比对待安装软件 vs 许可证 | 定时自动 |

**脚本安全模型**：

```
风险等级:
  Low (绿色)  → 只读操作，无需审批，自动执行
  Medium (黄色) → 配置变更，需用户确认后执行
  High (红色) → 系统级变更，需工程师审批后执行
```

### 2.4 持续优化层（Optimization）

**职责**：从运维数据中学习，持续改进。

| 优化方向 | 方法 | 周期 |
|---------|------|------|
| 知识库自动扩充 | 已解决工单 → LLM 摘要 → 草稿文章 → 人工审核发布 | 每日 |
| 自动化脚本迭代 | 执行失败分析 → 脚本修正建议 → 版本更新 | 按需 |
| AI 模型评估 | RAG 命中率、回答准确率、用户反馈评分 | 每周 |
| 故障预测模型 | 历史监控指标 + 故障标签 → XGBoost 训练 → 更新预测规则 | Phase 3 每月 |
| 运维流程优化 | 工单瓶颈分析 → 处理时长分布 → 推荐流程改进 | 每月 |

## 4. 数据流设计

### 4.1 监控数据流

```
Windows Endpoint → Zabbix Agent 2 (采集) → Zabbix Server → PostgreSQL (TimescaleDB 超表)
                                                           ↓
                                              FastAPI monitoring.ingest (备选直写路径)
                                                           ↓
                                              TimescaleDB: monitoring_metrics 超表
                                                           ↓
                                              FastAPI GET /monitoring/metrics
                                                           ↓
                                              React ECharts 实时面板
```

### 4.2 知识库数据流

```
管理员录入文章 → FastAPI POST /kb/articles
                  ↓
              Celery Worker: kb_tasks.index_article
                  ↓
              1. Markdown 分块 (512 tokens, overlap 50)
              2. LLMClient 嵌入 (默认 1024 维；降维或换模型需独立集合)
              3. Qdrant upsert (collection: kb_chunks)
                  ↓
用户搜索 → 嵌入查询文本 → Qdrant 语义检索 → Top-K 结果
                  ↓
            拼接 RAG Prompt → LLM Provider 生成回答
```

### 4.3 工单数据流

```
创建工单 (聊天机器人/手动/监控告警)
    ↓
状态: open → AI 诊断 (Celery 异步任务)
    ↓
L1 匹配? → 是 → 自动修复 → 状态: resolved → 自动提取至知识库
    ↓ 否
L2 匹配? → 是 → 生成诊断 + 触发脚本 → 状态: in_progress → resolved
    ↓ 否
L3 升级  → 指派工程师 → 状态: in_progress → 工程师处理 → resolved
    ↓
resolved → Celery Beat: kb_tasks.extract_from_ticket (自动提取至知识库)
```

## 5. 技术栈总览与选型理由

> **参见**：[DOC-17 技术栈全景方案](DOC-17-技术栈全景方案.md) — 含完整 10 层技术栈选型对比、替代方案分析、依赖版本矩阵。

```
│  ┌──────────────────────────────────────┐   │
│  │  JWT 认证中间件                        │   │
│  │  - Access Token: 30min               │   │
│  │  - Refresh Token: 7d                 │   │
│  │  - 存储在 httpOnly cookie            │   │
│  └──────────────────────────────────────┘   │
│  ┌──────────────────────────────────────┐   │
│  │  RBAC 授权                            │   │
│  │  - admin: 全权限                      │   │
│  │  - engineer: 工单处理、脚本执行、KB编辑│   │
│  │  - user: 自助服务、工单查看            │   │
│  └──────────────────────────────────────┘   │
└─────────────────────────────────────────────┘
```

### 6.2 数据安全

| 层面 | 措施 |
|------|------|
| 传输加密 | Nginx TLS 1.3 终止；Agent-Server 间 Phase 3 升级 mTLS |
| 存储加密 | PostgreSQL 透明数据加密 (TDE)；MinIO 服务端加密 |
| 敏感字段 | 许可证密钥、SNMP 社区串、API 密钥 → 数据库列级加密 (AES-256-GCM) |
| 密钥管理 | Phase 1: `.env` 文件 + Docker Secrets；Phase 3: HashiCorp Vault |
| 审计日志 | 所有 API 调用记录至 `audit_log` 表：user_id, action, resource, timestamp, ip, result |
| 网络隔离 | 全部服务在 Docker 内部网络通信；仅 Nginx 暴露宿主机端口 |

### 6.3 脚本执行安全

```
用户触发脚本 → 权限校验 → 风险等级检查 →
  Low:    自动执行
  Medium: 用户二次确认
  High:   工单审批 → 工程师确认 → 执行

执行隔离:
  - 所有脚本在工作目录执行 (非系统目录)
  - 脚本执行超时限制 (默认 300s)
  - 输出日志强制脱敏 (IP、密码、密钥)
```

## 8. 企业级模型部署路线

### 8.1 阶段策略

| 阶段 | 模型服务 | 预算策略 | 说明 |
|------|----------|----------|------|
| 开发 / PoC | 百炼 DeepSeek-V4 API | 按量付费，优先买速度 | 默认使用 `deepseek-v4-pro` + `text-embedding-v4`，先验证业务闭环 |
| 生产试点 | 百炼 DeepSeek-V4 专属实例或企业专线 API | 质量优先，控制并发峰值 | 对接企业网络出口、审计和密钥管理，沉淀真实评测集 |
| 正式生产 | DeepSeek-V4 私有化 GPU 推理集群 | 预算可上浮，按企业级容量规划 | 推荐 48GB/80GB 级别 GPU，支持多模型并发和后续多模态 |
| 站点应急 | 站点级中小模型 | 低成本冗余 | 网络隔离或中心模型不可用时提供基础问答和静态回退 |

### 8.2 GPU 容量规划口径

| 层级 | 推荐显存 | 适合模型 | 适用场景 |
|------|----------|----------|----------|
| 入门私有化 | 24GB | DeepSeek-R1-Distill 14B/32B 或 DeepSeek-V4 小规格量化 | 低并发试点、站点应急 |
| 企业标准 | 48GB | DeepSeek-V4-Flash / 蒸馏模型 / 多模型驻留 | 2000-5000 终端规模的常规生产 |
| 企业增强 | 80GB×2+ | DeepSeek-V4-Pro 或同级大模型 | 高并发诊断、复杂日志分析、未来多模态能力 |

**结论**：前期开发不采购 GPU，直接使用阿里云百炼 API；企业生产预算可适当提高，围绕模型质量、并发吞吐、审计合规和未来扩展规划 GPU，而不是以消费级 8GB 显卡作为上限。
