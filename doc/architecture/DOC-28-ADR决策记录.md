## ADR-001：模块化单体 vs 微服务

| 属性 | 值 |
|------|-----|
| **状态** | 已接受 |
| **日期** | 2026-06 |
| **决策者** | 架构团队 |

### 上下文

项目初期需要在开发速度和架构灵活性之间权衡。微服务虽然提供独立部署和扩展能力，但在团队规模小、终端数量可控的情况下，分布式系统带来的网络延迟、调试复杂度和运维负担远超收益。

### 决策

**Phase 1-2 采用模块化单体架构**。FastAPI 单进程部署，内部按模块（auth/assets/tickets/knowledge/chatbot/automation/monitoring/network/licenses/reports/ocr）划分清晰的代码边界。Phase 3 根据实际负载按需拆分关键模块为独立服务。

### 后果

- ✅ 开发速度快，本地调试简单，`docker compose up -d` 一键部署
- ✅ 模块间 service 层直接调用，无 HTTP 开销
- ⚠️ 需要在代码层面保持模块边界清晰（每个模块独立的 routes/service/models/schemas），确保未来可拆分
- ⚠️ 单体进程可能成为性能瓶颈——通过 Proxy 扩展层和 Celery Worker 池缓解
- 📅 触发拆分的条件：日活工程师 > 500 或终端数 > 5000

## ADR-002：DeepSeek-V4 开发测试与私有化路线

| 属性 | 值 |
|------|-----|
| **状态** | 已接受 |
| **日期** | 2026-06 |
| **决策者** | AI 团队 + 架构团队 |

### 上下文

需要选择开发期和生产期一致性较好的主力 LLM 路线。用户希望先在阿里云百炼上调用 API 快速测试，同时未来可私有化部署。DeepSeek-V4 Preview 已由官方发布并开源，百炼平台提供 DeepSeek-V4 调用入口，满足“开发期 API 测试、生产期私有化”的连续路线。

### 决策

**选择 DeepSeek-V4 作为主力模型路线**。开发期默认通过阿里云百炼 DashScope OpenAI 兼容接口调用 `deepseek-v4-pro`，常规 RAG、摘要和分类可使用 `deepseek-v4-flash` 控制成本。生产期优先基于 DeepSeek-V4 开源权重通过 vLLM 或企业统一模型网关私有化部署；资源受限或边缘站点可使用 DeepSeek-R1-Distill 系列兜底。

### 后果

- ✅ 开发期和私有化路线一致，减少模型迁移带来的 Prompt 与评测偏差
- ✅ 百炼 API 可以快速验证诊断质量，不被早期 GPU 采购阻塞
- ✅ DeepSeek-V4 开源后具备企业私有化部署基础
- ⚠️ 完整私有化 DeepSeek-V4 对 GPU 显存、推理框架和运维团队要求高
- ⚠️ `text-embedding-v4` 与 `BAAI/bge-m3` 默认均按 1024 维设计；若显式降维、替换模型或语义分布差异影响检索质量，必须新建 Qdrant collection 并重建索引

## ADR-003：TimescaleDB vs InfluxDB

| 属性 | 值 |
|------|-----|
| **状态** | 已接受 |
| **日期** | 2026-06 |
| **决策者** | 架构团队 |

### 上下文

监控时序数据（CPU、内存、磁盘等）需要专用时序存储。候选为独立部署 InfluxDB 或使用 PostgreSQL 扩展 TimescaleDB。

### 决策

**选择 TimescaleDB (PostgreSQL 扩展)**。减少运维一个独立数据库服务；时序数据可直接与业务数据 JOIN（如"查询 CPU > 90% 的资产所属部门"）；TimescaleDB 在单节点部署场景性能与 InfluxDB 相当。

### 后果

- ✅ 统一 PostgreSQL 运维，减少服务数
- ✅ 时序与业务数据 JOIN 能力
- ⚠️ 极大规模时序场景（千万点/秒）可能需要 InfluxDB，但本系统规模远未达到

- DOC-08（数据模型规格书）§3

## ADR-004：Qdrant vs pgvector

| 属性 | 值 |
|------|-----|
| **状态** | 已接受 |
| **日期** | 2026-06 |
| **决策者** | AI 团队 |

### 上下文

RAG 知识库需要向量数据库存储和检索文本嵌入。候选为 pgvector（PostgreSQL 扩展，零额外服务）和 Qdrant（专用向量数据库，需独立容器）。

### 决策

**选择 Qdrant 作为向量数据库**。Qdrant 的 HNSW 索引 + 元数据过滤性能优于 pgvector；支持量化压缩减少内存；单容器部署运维简单。虽然增加一个独立服务，但额外 50-100MB 内存开销可接受。

### 后果

- ✅ 向量检索性能最优（HNSW + 量化压缩）
- ✅ 元数据过滤灵活
- ⚠️ 增加一个独立服务，增加运维负担（可接受）
- 📅 如果未来简化部署需求强烈，可评估迁移至 pgvector

## ADR-005：钉钉作为首要 IM

| 属性 | 值 |
|------|-----|
| **状态** | 已接受 |
| **日期** | 2026-06 |
| **决策者** | 产品团队 |

### 上下文

IntelliServe 需要选择首要 IM 入口。候选为钉钉和企业微信。内网部署场景下，企业微信需要公网回调 URL，与"数据不出企业内网"的核心原则冲突。

### 决策

**选择钉钉作为首要 IM 入口**。钉钉 Stream Mode 使用 WebSocket 长连接，无需公网回调地址，天然适合内网部署场景。同时钉钉提供原生审批 API、交互卡片消息，更适合 IT 运维场景。企业微信作为次要通道（Phase 2 支持）。

### 后果

- ✅ 内网部署无需公网 IP/域名
- ✅ 原生审批流集成（高风险脚本审批）
- ✅ 交互卡片提供丰富 UI 能力
- ⚠️ 企业微信用户（中国市场占有率高）需等待 Phase 2 支持
- ⚠️ 需要维护两套 IM 适配层

- DOC-06（Webhook 与回调规范）
- DOC-24（钉钉集成详细设计）

## ADR-006：2000-5000 台规模定位变更

| 属性 | 值 |
|------|-----|
| **状态** | 已接受 |
| **日期** | 2026-06 |
| **决策者** | 产品团队 + 架构团队 |

### 上下文

原始方案定位于 50-500 台终端的中小型团队。经过需求评审，目标场景升级为 2000-5000 台终端的大型园区。这影响架构设计、部署方案、硬件配置和 Proxy 架构。

### 决策

**将系统目标规模升级为 2000-5000 台终端**。架构上通过多 VLAN Proxy 层实现水平扩展，而非早期拆分微服务。核心服务保持模块化单体，Proxy 层处理分区隔离和流量汇聚。

### 后果

- ✅ 架构具备弹性：通过 Proxy 层扩展而非重构核心
- ⚠️ 需要为每个 VLAN 部署 Proxy 实例
- ⚠️ 数据库需要读写分离（主从复制）
- ⚠️ Celery Worker 需要集群化（3-5 节点）
- ⚠️ 时序数据 90 天存储量从 ~50GB 增长至 ~500GB
- 📅 微服务拆分阈值从 1000 台上调至 5000 台

## ADR-007：Zabbix Agent + 自研 Agent 混合方案

| 属性 | 值 |
|------|-----|
| **状态** | 已接受 |
| **日期** | 2026-06 |
| **决策者** | 架构团队 |

### 上下文

终端 Agent 需要在"快速落地"和"功能完备"之间权衡。纯 Zabbix Agent 2 方案成熟但缺乏远程命令执行和本地诊断能力。纯自研 Agent 方案功能完备但开发周期长（8-12 周）。

### 决策

**采用渐进式混合方案**。Phase 1 全量部署 Zabbix Agent 2 快速获得基础监控和资产数据（1-2 周）；Phase 2 开发自研 Go Agent 在 IT 部门试点（4-8 周）；Phase 3 逐步替换为统一的自研 Agent。

### 后果

- ✅ Phase 1 即可获得 2000+ 终端的完整监控数据
- ✅ 两套 Agent 互补（Zabbix 成熟 + 自研灵活）
- ⚠️ Phase 2 期间需要同时运维两套 Agent（额外 30-60MB/终端）
- ⚠️ 最终迁移到纯自研 Agent 需要完善的升级和回滚机制
- 📅 全量替换 Zabbix Agent 的目标时间：Phase 3 中期
